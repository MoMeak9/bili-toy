# 本地音频实验室需求补齐 Todo

- 日期：2026-06-07
- 范围：MVP + 二阶段完成后的发布补齐与 Phase 3 候选池
- 来源：`docs/PRD.md`、二阶段 spec、二阶段实现审计、浏览器 QA 结果
- 状态：需求补齐清单，不直接代表实现计划

## 1. 背景

本地音频实验室已经完成 MVP 与二阶段核心能力：上传、录音入口、示例音频、波形、播放、P0 + V1.5 预设、效果链展示、基础分析、频谱图、WAV / MP3 导出，以及 Tone / wavesurfer / MP3 / 频谱模块懒加载。

本文用于补齐两类工作：

1. 当前版本发布前必须完成或建议完成的验收、稳定性和文档事项。
2. 不阻塞当前发布、但应进入后续规划的 Phase 3 候选能力。

## 2. 当前完成基线

### 2.1 已完成能力

- SPA 单页形态：核心流程不跳转页面。
- 首页三 CTA：上传音频、开始录音、打开示例。
- 上传 / 示例统一进入编辑器。
- 录音面板、麦克风权限请求、MediaRecorder 录制、录音转 `AudioBuffer`。
- 波形展示、播放 / 暂停、A/B 对比。
- P0 预设：机器人、魔鬼低音、松鼠音、电话音、广播主持。
- V1.5 预设：外星人、磁带机、山洞回声、8-bit。
- 效果链面板：展示当前预设处理链，并支持重置为无预设。
- Inspector 参数：音量、变速、变调、淡入、淡出。
- 专业分析：时长、采样率、声道数、Peak、RMS、频谱图。
- 导出增强：WAV / MP3、导出状态、进度、错误提示和重试入口。
- 性能整理：Tone、wavesurfer、MP3 编码器、频谱模块按需加载。
- 移动端：预设、效果、参数、分析四个底部 Tab。
- README 已覆盖本地处理、无后端、无 AI、快速开始和核心能力。

### 2.2 已验证命令

当前 `master` 上已验证：

```bash
npx tsc --noEmit
npm run lint
npm run build
```

构建输出显示首屏主入口约 `296 kB`，并拆出 `spectrum`、`mp3Encoder`、`wavesurfer.esm`、Tone 相关独立 chunk。

### 2.3 已知人工验证缺口

以下事项需要真实浏览器权限、真实下载文件或多浏览器环境，不能仅依赖当前自动化浏览器验证：

- 麦克风权限允许后的完整录音流程。
- 麦克风权限拒绝后的恢复流程。
- WAV / MP3 实际下载文件可播放。
- 导出异常路径的真实恢复体验。
- Safari / Edge / 移动端浏览器兼容性。

## 3. P0 发布阻断 Todo

P0 表示不完成就不应对外宣称当前版本可发布。

- [ ] 真实麦克风权限允许路径验证
  - 来源：PRD `开始录音`、二阶段录音闭环 spec。
  - 验收：用户点击开始录音，授权麦克风，录制 3 秒以上，停止后进入编辑器；波形、播放、预设、分析、WAV / MP3 导出入口仍可用。
  - 备注：需要在真实 Chrome 或 Edge 中手动授权麦克风完成。

- [ ] 真实麦克风权限拒绝路径验证
  - 来源：PRD `除浏览器麦克风录音必须授权外，其余功能都应尽量直接可用`、二阶段错误恢复策略。
  - 验收：用户拒绝麦克风权限后看到可理解错误；应用不白屏；用户能取消或返回 landing；再次进入录音流程时行为可预测。
  - 备注：需要浏览器权限弹窗或站点权限设置辅助验证。

- [ ] WAV 下载文件可播放验证
  - 来源：MVP WAV 导出验收、二阶段导出增强验收。
  - 验收：示例音频或录音素材导出 WAV 后，本地播放器或浏览器可以播放；时长与源素材近似一致；已选择预设时导出结果包含处理效果。
  - 备注：自动化浏览器不能稳定接管本机下载，需要人工下载并播放。

- [ ] MP3 下载文件可播放验证
  - 来源：二阶段导出增强完成定义。
  - 验收：选择 MP3 后导出并下载 `.mp3` 文件；本地播放器或浏览器可以播放；文件不是 0 字节；时长与源素材近似一致。
  - 备注：必须验证 `lamejs` 产物在目标浏览器环境中可播放。

- [ ] 导出失败恢复路径验证
  - 来源：二阶段导出增强 spec、错误恢复策略。
  - 验收：模拟或触发导出失败后，当前编辑器项目、参数、预设不丢失；导出弹窗显示错误；用户能关闭弹窗或重试。
  - 备注：可以通过临时断点、抛错开关或手动 monkey patch 编码函数验证，验证后不得留下调试代码。

- [ ] 移动端核心闭环验证
  - 来源：PRD 移动端开箱即用、二阶段完成定义。
  - 验收：移动 viewport 或真机中，首页三 CTA 可见；录音入口可达；示例进入编辑器；预设 / 效果 / 参数 / 分析 Tab 可访问；导出弹窗可操作。
  - 备注：当前已做自动化 viewport QA，仍建议用一台真实移动设备或移动浏览器补测。

## 4. P1 发布前建议 Todo

P1 不阻断当前功能完成，但会提升发布可信度、用户理解和稳定性。

- [ ] 首次进入编辑器快捷键提示
  - 来源：PRD `首次提示`。
  - 验收：用户首次从上传、录音或示例进入编辑器时看到一次 Toast：`提示：按 Space 播放/暂停，按 ? 查看全部快捷键。`；同一会话内不重复打扰。
  - 备注：可用内存状态或 localStorage 控制展示频率。

- [ ] README 增加浏览器兼容与权限说明
  - 来源：录音闭环风险、PRD 开箱即用要求。
  - 验收：README 明确说明推荐 Chrome / Edge；Safari 录音格式可能不同；麦克风权限只用于本地录音；所有音频不上传。
  - 备注：不要写成过度承诺的兼容矩阵，先描述已验证环境和建议环境。

- [ ] 录音格式兼容说明
  - 来源：MediaRecorder 格式差异风险。
  - 验收：UI 或 README 说明录音文件名扩展名只是来源标记，实际录音编码由浏览器支持的 mimeType 决定；解码失败时提示用户换浏览器或重试。
  - 备注：当前实现优先 `audio/webm;codecs=opus`、`audio/webm`、`audio/mp4`。

- [ ] 导出弹窗补充 MP3 首次加载说明
  - 来源：性能整理、导出增强。
  - 验收：用户选择 MP3 时能看到短文案说明首次导出会加载本地编码器，可能稍慢；导出仍然不上传音频。
  - 备注：文案应短，不新增教学式说明页。

- [ ] 手测矩阵文档化
  - 来源：二阶段测试策略。
  - 验收：新增或更新文档，记录 Chrome / Safari / Edge、桌面 / 移动、上传 / 录音 / 示例 / 导出 / 分析的手测结果。
  - 备注：可写入 `docs/superpowers/qa/` 或 README 的简短章节。

- [ ] 键盘路径回归验证
  - 来源：PRD 桌面快捷键、可访问性基础要求。
  - 验收：`Space` 播放 / 暂停、`?` 打开快捷键弹窗；Tab 能到达主要按钮；Dialog 关闭后焦点行为不阻塞继续使用。
  - 备注：不要求完整 WCAG 审计，但要验证核心键盘路径。

## 5. P2 发布后优化 Todo

P2 可以在当前版本发布后继续迭代，但不应丢失追踪。

- [ ] 录音过程中加入音量指示
  - 来源：录音体验优化。
  - 验收：录音中展示简单输入电平或动画，让用户知道麦克风正在收到声音。
  - 备注：二阶段不做实时波形，P2 可先做轻量电平条。

- [ ] 导出错误分类
  - 来源：导出失败恢复体验。
  - 验收：导出失败提示能区分离线渲染失败、MP3 编码失败、下载触发失败三类。
  - 备注：保持用户文案简洁，调试信息可进入 console。

- [ ] 分析面板 loading / 空状态优化
  - 来源：专业分析体验。
  - 验收：频谱生成中有明确 loading；空 buffer 或计算失败有可恢复提示；重复点击不会并发计算。
  - 备注：当前已有基础错误提示，可继续 polish。

- [ ] 首屏性能预算记录自动化
  - 来源：性能整理。
  - 验收：提供一个脚本或文档步骤，记录主 chunk、Tone chunk、wavesurfer chunk、MP3 chunk、频谱 chunk 体积。
  - 备注：可先用 `npm run build` 输出人工记录，不必立刻接入 CI。

- [ ] 无障碍细节补齐
  - 来源：SPA 工具可用性。
  - 验收：关键 icon-only 按钮有 aria-label；导出格式 segmented control 的当前选择可被辅助技术识别；录音和导出状态变化有可读文本。
  - 备注：避免大规模设计系统重构。

## 6. Phase 3 候选池

Phase 3 候选池不承诺当前发布范围。进入实现前应单独写 spec 和 implementation plan。

### 6.1 PWA / 离线

- [ ] PWA 离线缓存
  - 来源：PRD `无网络：PWA 缓存后可打开应用并处理本地文件`。
  - 验收：首次访问后，用户断网仍能打开应用、上传本地文件、使用已缓存的核心编辑能力。
  - 备注：需要明确缓存策略，避免错误缓存大音频导出产物。

- [ ] 安装到桌面 / 主屏幕体验
  - 来源：PWA 产品化。
  - 验收：应用具备 manifest、图标、主题色；支持安装；打开后仍保持 SPA 工具体验。
  - 备注：不引入账号、后端或云同步。

### 6.2 导出增强

- [ ] OGG 导出
  - 来源：PRD 导出编码候选。
  - 验收：导出弹窗新增 OGG；编码库按需加载；导出文件可播放。
  - 备注：需要先评估浏览器端编码库体积与兼容性。

- [ ] 分段导出
  - 来源：PRD 长期导出能力。
  - 验收：用户可以选择起止时间并只导出片段；未选择时仍默认整段。
  - 备注：需要先设计选区 UI 和波形交互。

- [ ] 批量导出
  - 来源：PRD 长期导出能力。
  - 验收：用户可对多个预设或多个格式批量导出。
  - 备注：可能显著增加 UI 和性能复杂度，应晚于分段导出。

### 6.3 录音增强

- [ ] 输入设备选择
  - 来源：录音体验增强。
  - 验收：用户可选择麦克风设备；无设备 label 权限时提示需先授权。
  - 备注：浏览器权限模型会影响设备名展示。

- [ ] 录音实时波形
  - 来源：录音体验增强。
  - 验收：录音过程中显示实时波形或近实时电平曲线。
  - 备注：比 P2 电平条复杂，需要 AudioWorklet 或 AnalyserNode 设计。

- [ ] 录音倒计时 / 最大时长
  - 来源：移动端和稳定性。
  - 验收：用户可看到录音时长上限；达到上限自动停止并进入处理。
  - 备注：可防止超长录音导致内存压力。

### 6.4 效果系统

- [ ] 效果链排序
  - 来源：PRD 长期效果能力。
  - 验收：用户可调整效果节点顺序，A/B 与导出结果一致。
  - 备注：需要从 preset-only 模型升级到 user chain 模型。

- [ ] 单效果开关
  - 来源：效果链可视化后续。
  - 验收：用户可临时关闭链中某个效果；关闭后播放和导出一致。
  - 备注：需要设计效果节点状态序列化。

- [ ] 用户自定义预设保存
  - 来源：项目保存和效果系统增强。
  - 验收：用户可以保存当前参数和效果链为本地预设，并在下次打开应用时使用。
  - 备注：依赖 localStorage 或 IndexedDB。

### 6.5 分析能力

- [ ] LUFS-lite 响度分析
  - 来源：专业分析增强。
  - 验收：分析面板展示近似 LUFS 或更专业的响度指标，并说明其为浏览器端近似值。
  - 备注：不承诺广播级完整合规，避免误导。

- [ ] 峰值定位
  - 来源：编辑辅助。
  - 验收：用户能看到最大峰值所在时间点，并可点击跳转。
  - 备注：需要和 waveform seek 交互结合。

- [ ] 分析报告导出
  - 来源：专业分析长期能力。
  - 验收：用户可导出 JSON 或文本摘要，包含基础信息和频谱采样结果。
  - 备注：不影响音频导出流程。

### 6.6 项目保存

- [ ] 最近文件 / 最近项目
  - 来源：本地工具产品化。
  - 验收：用户可看到最近编辑过的项目元数据，并重新打开仍保留参数。
  - 备注：浏览器不能持久保存原始文件路径，需设计本地数据边界。

- [ ] 本地项目保存
  - 来源：长期项目能力。
  - 验收：用户可保存当前音频、参数、预设、分析结果到本地浏览器存储。
  - 备注：需要 IndexedDB，且要给出清除数据入口。

### 6.7 工程质量

- [ ] E2E 测试覆盖核心路径
  - 来源：发布稳定性。
  - 验收：自动化覆盖首页、示例进入编辑器、播放按钮状态、导出弹窗、移动 tabs。
  - 备注：麦克风和下载仍需要专门 mock 或人工验证。

- [ ] 音频核心单元测试
  - 来源：导出和分析稳定性。
  - 验收：`floatToInt16`、WAV 编码、频谱计算、summary 计算有可重复测试。
  - 备注：需要引入测试 runner 或使用现有构建工具能力。

- [ ] 浏览器兼容测试表
  - 来源：录音和导出风险。
  - 验收：记录 Chrome、Edge、Safari 桌面和移动浏览器的上传、录音、导出、分析结果。
  - 备注：作为发布记录，不必自动化。

## 7. 不纳入当前补齐范围

以下内容不进入 P0 / P1 发布补齐：

- 后端上传、账号、云项目。
- AI 音频处理。
- 多轨编辑。
- 完整 DAW 级参数自动化。
- 云同步。
- 复杂协作或分享链接。
- 付费、套餐、登录或用户体系。

## 8. 推荐执行顺序

1. 先完成 P0 人工验收，确认当前版本可以被称为可发布版本。
2. 再完成 P1 文档、提示和兼容性说明，降低用户误解和发布风险。
3. P2 作为发布后短周期优化池，优先选体验成本低、收益明确的条目。
4. Phase 3 候选池进入下一轮 brainstorming，单独挑选范围后再写 spec 和 implementation plan。

## 9. 后续计划入口

如果要进入实现计划，建议先只选择：

- P0 发布阻断全部条目。
- P1 中 README、快捷键提示、手测矩阵三项。

Phase 3 候选池应在当前发布补齐完成后再单独规划。
